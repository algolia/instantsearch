Findings — trigger, backend condition, handoff, context
  • Backend condition: the idiomatic Algolia mechanism already exists — a Rule with a userData consequence, read from results.userData on the frontend. The demo index has no such Rule, so this POC evaluates a simulated rules table in the same shape (and honors real userData.chatResultCard if the index ever returns it, including a position override). No new backend API needed for a first version.
  • Follow-up → full chat: works today with public APIs: copy the card's messages into the main chat via setMessages(), then setOpen(true). Two caveats found while building: (1) renderState[…].messages is stale during streaming (chat re-renders don't refresh the shared snapshot), so the POC captures live messages from the card's own render callback — a productized handoff API should hide this; (2) conversation ids differ between the two instances — the stateless transport makes it work anyway, but analytics/feedback attribution would want a proper "transfer conversation" concept.
  • PLP context: connectChat already supports it via the context option → metadata.turnContext on the user turn. The Agent Studio backend accepts it and demonstrably uses it (try the compare lab). Caching note: identical questions may be served from the completion cache; the compare lab bypasses it with cache=false.
  • Is auto-trigger good enough? Search-as-you-type means the trigger must be debounced (700 ms here) and deduplicated per question, or you spam the agent. A backend condition (Rule) is the cleanest gate because merchandisers control it per query/context, and it can also carry the card position.
